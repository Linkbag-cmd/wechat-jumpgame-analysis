地图随机种子。

客户端和服务器用同一个 seed 生成完全相同的方块序列（位置、颜色、类型、音乐块、自动块等）。

作用：防止我们改地图或直接改分数——服务器会用这个 seed 重建一模一样的地图，再看操作能不能跳到 198 分。

time_seed: 1772598953405

时间基准种子（毫秒级时间戳）。

所有 timestamp 都是相对这个时间算的，防篡改时间轴或加速/减速脚本。

action: [[0.69,1.33,false], [0.706,1.33,false], ...] （约 59-60 条记录）

每一次跳跃的操作参数（最核心的反作弊数据）。

每个子数组格式：[x, t, b]

x：手指按压位置的横向比例（0~1，屏幕宽度归一化）。0.69 就是屏幕左到右的 69% 位置。决定跳跃方向/角度。

t：按压时长（秒）。1.33 秒 = 按住了 1.33 秒，决定跳跃距离（越长跳越远）。

b：boolean（false/true），通常表示是否长按特殊模式 或 是否触发完美跳跃/道具。


从这里我们也就能看出服务器的验证逻辑了：

1.用 seed 重建一模一样的地图。

2.用 action + touchList + timestamp 逐帧重放小人的物理轨迹。

3.检查最终落地点数量、得分、音乐块触发是否和提交的 score 完全一致。

4.检查行为是否「像人」：按压位置抖动、时间间隔自然度、指压时长分布等。

5.校验 mmpay_checksum 和其他签名。

_3.思考：为什么2026年的跳一跳几乎不可能伪造高分_

经过对多局抓包数据的拆解，我们可以看到腾讯在《跳一跳》这个“老古董”小程序上，并没有简单粗暴地升级加密算法（actionData 依然是2018年的 AES-CBC + session_id[:16] 套路），而是把防御重心彻底后移，构建了一套多层、交叉、难绕的铁桶阵。

核心检测链路总结如下 ：
第一层：格式与基本校验

actionData 必须用紧凑 JSON（无空格、无换行）加密，否则 base64 解密后不匹配直接拒。

第二层：种子重放 + 逐帧物理仿真

seed 和 time_seed 是核心。服务器会用相同的种子重跑你的 action 数组，模拟每一次跳跃的物理轨迹（重力、速度、碰撞、落地角度等）。哪怕把 score 改成 9999，只要 action 序列和物理结果对不上（落地位置偏差、速度异常），立即判假。
这也是为什么单纯复制高分局的 action 数组也过不了——种子不同，重放就崩。

第三层：行为统计与 AI 画像

touchList 坐标分布：人类有微抖动（±几像素），脚本常固定（如你这局全 203.5,430.5）。

true/false 比例与切换规律：人类更随机，脚本常出现规律性连跳或间隔。

timestamp 间隔：人类近似高斯分布（均值~1.2-1.5s，标准差大），脚本太均匀或太极端。

整体时长与分数匹配度：76秒跳59次 + score 198，已偏离正常人类曲线。


第四层：多接口交叉校验

wxagame_bottlereport（终局统计：duration、using_prop、break_record）、eggreport（道具/金蛋）、adbanner（广告链路）等接口会把同一局的多个维度数据上报。改一处（如 score），其他接口（如 mmpay_checksum、report_list type 35/10/2）不匹配 → 风控秒触发。

第五层：mmpay_checksum 与设备指纹

这个字段（局里是 "d6900e074f0895c36de229b7f1cad46a"）很可能基于整包数据 + 盐 + 设备信息做 MD5/HMAC 签名。伪造它需要知道腾讯内部盐值，几乎不可能。加上 client_info 的设备指纹（IP、UA、微信版本、硬件等），同一设备短时多次异常提交 = 永久标记。


_ 4.结语：考古的意义与现实提醒_

《跳一跳》虽是2017年的老游戏，但腾讯用“换汤不换药”的方式，让它在2026年依然保持着极高的安全壁垒。这套机制对其他微信小游戏（乃至小程序生态）都有借鉴价值：加密只是表层，真正的防护在于数据一致性、物理真实性与行为自然度的多维校验。
